# Contract: Project-Scale Delivery UX

This contract describes the user-facing CLI and assistant behavior required by the feature. Exact CLI option spelling may be refined during implementation, but the semantics are normative.

## Global Assistant Install Contract

```bash
boundline assistant install --host claude --scope user
boundline assistant install --host codex --scope user
boundline assistant install --host cursor --scope user
boundline assistant install --host copilot --scope user
boundline assistant install --host gemini --scope user
```

**Required behavior**:

- Validates the requested host.
- Reports whether true user-scoped/global commands are supported.
- Installs the package when supported or prints exact manual/CLI fallback instructions when unsupported.
- Does not require `.boundline/session.json`.
- Writes or references package metadata whose version matches the Boundline package version.

## Global Command Contract

Global assistant packages must expose:

- `/boundline:init`
- `/boundline:doctor`
- `/boundline:help`
- `/boundline:continue`
- `/boundline:status`

They may expose `/boundline:deliver` and `/boundline:govern` only when behavior remains safe without repo-local state.

For every command:

- If shell execution is available, the command invokes the corresponding Boundline CLI surface.
- If shell execution is unavailable, the command shows exact CLI commands.
- If workspace is unknown, the command asks for or detects the workspace.
- If `.boundline/session.json` is absent, the command does not infer session state from chat history.
- The command must end with a concrete next step.

## Repo-Local Command Contract

Repo-local commands generated by `boundline init --assistant <host>` must:

- Preserve the session-native loop.
- Read state through Boundline CLI and `.boundline/session.json`.
- Surface `active`, `blocked`, `clarification_required`, `failed`, `exhausted`, and `terminal` states explicitly.
- Show the same next action as CLI `status`, `next`, or `inspect`.
- Avoid duplicating runtime logic in markdown command bodies.

## Project-Scale Path Contract

When a user asks Boundline to deliver a broad idea, Boundline must return a proposed path object equivalent to:

```json
{
  "goal": "Build a customer onboarding capability with audit logging.",
  "path": [
    {"stage": "discovery", "reason": "Problem framing is incomplete"},
    {"stage": "requirements", "reason": "Acceptance and scope must be bounded"},
    {"stage": "system-shaping", "reason": "Capability boundaries are not fixed"},
    {"stage": "architecture", "reason": "Audit logging has structural implications"},
    {"stage": "backlog", "reason": "Delivery slices must be decomposed"},
    {"stage": "implementation", "slice": 1},
    {"stage": "verification"},
    {"stage": "pr-review"}
  ],
  "requires_confirmation": true,
  "next_action": "confirm_path"
}
```

The path must be editable by explicit confirmation, clarification, or replanning. It must not imply one unchecked autonomous run.

## Govern Contract

Assistant surface:

```text
/boundline:govern
/boundline:govern architecture
/boundline:govern security-assessment
/boundline:govern pr-review
```

CLI equivalent:

```bash
boundline govern --mode architecture --brief tech-docs/architecture.md
boundline govern --mode requirements --goal "Bound the customer onboarding capability"
boundline govern --mode pr-review --base refs/heads/main --head HEAD
```

**Required behavior**:

- Validates requested mode against Canon capabilities.
- Infers and presents choices when no mode is provided.
- Requires confirmation when a material or high-risk governed stage is proposed.
- Stops when required input shape, system context, Canon compatibility, approval state, or readiness is missing.
- Persists governed packet refs, provenance refs, approval state, and next action in Boundline session state.
- Does not promote primary `/boundline-<mode>` aliases.

## Canon Capability Contract

Before running governed stage work, Boundline must obtain a capability snapshot equivalent to:

```json
{
  "canon_version": "0.45.0",
  "modes": [
    "discovery",
    "requirements",
    "system-shaping",
    "architecture",
    "backlog",
    "change",
    "implementation",
    "refactor",
    "review",
    "verification",
    "pr-review",
    "incident",
    "security-assessment",
    "system-assessment",
    "migration",
    "supply-chain-analysis"
  ]
}
```

Unavailable, missing, malformed, or incompatible capability output must block governed execution with repair guidance.

## Voting State Contract

Status, next, and inspect surfaces must be able to project:

```json
{
  "voting": {
    "trigger": "validation_exhausted",
    "reviewed_evidence_ref": "trace:...",
    "strategy": "reject_on_blocking",
    "result": "blocked",
    "findings": [
      {"reviewer": "security", "vote": "block", "summary": "Auth boundary change lacks coverage"}
    ],
    "adjudication": null,
    "blocking": true,
    "next_action": "adjudicate_or_repair"
  }
}
```

Low-risk stages must omit or explicitly mark voting as skipped by policy unless the operator requested it.

## Status/Inspect Parity Contract

For the same workspace and active session, CLI and assistant surfaces must agree on:

- current session id
- current stage and work unit
- next action
- governance mode and packet refs
- approval/readiness state
- voting state and blocking result
- trace refs
- checkpoint refs
- exact CLI-equivalent next command
