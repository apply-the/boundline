# Research: Boundline Project-Scale Delivery UX

## Decision: Split Global And Repo-Local Assistant Packages

Boundline should define two assistant package scopes. The global or user-scoped package is installed once per host and exposes bootstrap-safe commands (`/boundline:init`, `/boundline:doctor`, `/boundline:help`, `/boundline:continue`, `/boundline:status`, plus safe deliver/govern guidance where possible). The repo-local package remains generated by `boundline init --assistant <host>` and owns workspace-specific command bindings, prompts, defaults, and metadata.

**Rationale**: Repo-local commands cannot solve first-run bootstrap because they do not exist before initialization. A global package can discover or ask for a workspace, detect whether `.boundline/session.json` exists, and route to exact CLI guidance without pretending chat memory is authoritative.

**Alternatives considered**:

- Only repo-local packages: rejected because uninitialized repositories remain undiscoverable from chat.
- Only global packages: rejected because workspace-specific defaults and state-aware command bindings still belong near the repository.
- Chat-only bootstrap instructions: rejected because the product needs a concrete installable or documented host package path where hosts support it.

## Decision: Do Not Pretend Universal Host Global Installation Exists

Host support must be represented exactly. Claude Code, Codex, Cursor, Copilot-style prompt environments, and Gemini-style CLI/chat environments may each have different user-scoped command capabilities. Where true global command installation is unavailable or uncertain, docs must say so and provide the closest supported import, prompt-pack, extension-ready metadata, or CLI-first path.

**Rationale**: The feature depends on trust in Boundline's operator guidance. Claiming unsupported host capabilities would create brittle install paths and conflict with the spec boundary that normal users must not hand-edit manifests.

**Alternatives considered**:

- Normalize every host to the same manifest shape: rejected because provider support differs.
- Hide unsupported hosts until later: rejected because the prompt explicitly asks for supported-host handling and honest fallbacks.

## Decision: Use One Governed Stage Catalog For The Full Canon Mode Set

Boundline should maintain a governed stage catalog covering discovery, requirements, system-shaping, architecture, backlog, change, implementation, refactor, review, verification, pr-review, incident, security-assessment, system-assessment, migration, and supply-chain-analysis.

**Rationale**: The catalog lets Boundline know when to consider each Canon mode, what system context is required, whether voting may apply, whether the stage can lead to implementation/refactor, and whether Canon output is recommendation-only. This keeps Canon at governed stage boundaries while Boundline owns orchestration.

**Alternatives considered**:

- Hard-code only implementation/change/review modes: rejected because project-scale delivery needs discovery through operational/risk paths.
- Expose a top-level command per Canon mode: rejected because that makes Boundline look like a Canon alias layer.
- Let Canon choose the entire path: rejected because Canon governs packets, while Boundline drives delivery.

## Decision: `/boundline:govern` Is The Single Primary Governed Stage Surface

Explicit governed stage work should route through `/boundline:govern` in assistant hosts and a CLI-equivalent surface such as `boundline govern --mode <mode> ...`. When no mode is provided, Boundline can infer likely modes or show supported choices. When a mode is specified, Boundline validates it against `canon governance capabilities --json` or the equivalent configured capability source before proceeding.

**Rationale**: A single governed surface preserves Boundline's product identity, supports the full Canon mode set, and gives users a clear mental model for stage-boundary governance.

**Alternatives considered**:

- `/boundline-architecture`, `/boundline-backlog`, and similar aliases: rejected as primary UX because they fragment the command set and obscure Boundline's orchestration role.
- Manual Canon packet manifests: rejected for normal operation because the feature requires no hand-edited JSON or manifests.

## Decision: Voting Is Risk/Evidence-Triggered Stage-Boundary Quality Control

Voting should be triggered by material risk or evidence gaps: high-impact architecture, Type 1 decisions, high-risk change boundaries, public contract/API/auth/schema/integration/data ownership changes, validation exhaustion, PR readiness, material security findings, high/critical supply-chain findings, migration cutover decisions, and high-blast-radius incident follow-up. Voting should not run by default for every discovery, requirements, low-risk local change, or low-risk preserved-behavior refactor.

**Rationale**: Voting is valuable when one reviewer or model is insufficient. Making it universal would slow delivery and violate the bounded sequential model.

**Alternatives considered**:

- Vote on every governed packet: rejected as too heavy and not delivery-first.
- Never vote by default: rejected because project-scale delivery needs a credible escalation path for risky boundaries.

## Decision: Delivery Pilot Model Belongs In Dedicated Docs And Architecture Cross-Links

Add `docs/delivery-model.md` and cross-link it from `docs/architecture.md`, `docs/getting-started.md`, and `README.md`. The docs should define "Large work is supported by decomposition, not by unbounded autonomy" and explain `observe -> decide -> act -> verify -> update context`.

**Rationale**: The model is product-facing and architectural. A dedicated doc prevents the architecture page from becoming overloaded while still giving users and contributors an obvious reference.

**Alternatives considered**:

- Only update `docs/architecture.md`: rejected because the user requested a visible model section and project-scale examples.
- Only update README: rejected because the behavior needs enough detail for contributors and testers.

## Decision: Version And Canon Compatibility Should Move First In Implementation

The first implementation task should improve the Boundline version, likely to the next minor feature version (`0.50.0` from current workspace version `0.49.1`) unless release management chooses a different value, and align Cargo, lockfile, distribution metadata, assistant metadata, docs, and tests. Canon compatibility target should be `0.45.0`.

**Rationale**: The user explicitly requires the first task to improve the Boundline version, and this slice introduces a visible project-scale UX. Aligning distribution metadata first avoids the stale-version gap previously seen in winget manifests.

**Alternatives considered**:

- Delay versioning until release: rejected by explicit user instruction.
- Leave old distribution folders untouched: rejected because version consistency is part of the requested review discipline.

## Decision: Model Catalog Requires No Entry Change For This Implementation Slice

Public provider model docs were checked on 2026-05-11 and refreshed again on 2026-05-12:

- OpenAI Models: https://platform.openai.com/docs/models
- Anthropic Claude Models overview: https://docs.anthropic.com/en/docs/about-claude/models/overview
- Google Gemini API Models: https://ai.google.dev/gemini-api/docs/models
- GitHub Copilot supported models: https://docs.github.com/en/copilot/reference/ai-models/supported-models

The current `assistant/catalog/model-catalog.toml` already includes the active model families needed for this feature's routing decisions, and this slice does not introduce new model-routing behavior. Implementation updated the catalog metadata version/date to `0.50.0` / `2026-05-12` but did not add or remove model entries.

**Rationale**: The feature changes delivery UX, command packaging, governance routing, voting state, and docs. It does not require new model entries or default routing changes.

**Alternatives considered**:

- Update catalog opportunistically: rejected because no catalog delta is justified by the reviewed docs for this slice.
- Skip catalog research: rejected by the Boundline Spec Kit constitution.
