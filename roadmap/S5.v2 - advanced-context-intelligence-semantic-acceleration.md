# Advanced Context Intelligence V2 — Semantic Acceleration

## Status

Future Research Specification

## Priority

After S5 V1, Before S6 And S7

This specification is intentionally deferred until:

- the S5 V1 local SQLite + FTS5 retrieval baseline is operational
- authority ordering is visible on normal runtime surfaces
- structured-only fallback and degraded retrieval paths are inspectable
- the workspace-local retrieval index lifecycle is stable

This specification MUST NOT replace S5 V1.

S5 V1 remains the required retrieval baseline.

S5.v2 extends that baseline with optional semantic acceleration on the same
workspace-local SQLite index.

---

# 1. Outcome

Add optional local semantic acceleration to Advanced Context Intelligence by
layering `sqlite-vec` and local embeddings on top of the existing SQLite
retrieval store.

The goal is to improve:

- semantic similarity matching
- implementation precedent retrieval
- review finding recall
- Canon artifact similarity matching
- evidence discovery quality

The goal is NOT to create a second retrieval architecture.

---

# 2. Product Thesis

S5 V1 provides the required local-first retrieval baseline through structured
indexes and FTS-backed search.

That baseline should ship first because it already delivers bounded,
explainable value.

S5.v2 exists to improve recall and similarity after the baseline is trusted.

If `sqlite-vec` is disabled, unavailable, or unsupported in a workspace,
Boundline must continue operating on the S5 V1 path without feature loss in the
core delivery loop.

---

# 3. Architectural Boundary

## 3.1 V1 Remains The Base Contract

The following remain true after S5.v2 lands:

- structured runtime context remains authoritative
- Canon project memory remains semantic input, not runtime control
- the FTS-backed local retrieval path remains required
- semantic acceleration remains optional

S5.v2 MUST NOT change the authority order:

```text
Structured Runtime Context
→ Canon Governed Memory
→ Workspace Overrides
→ Semantic Retrieval
→ Similarity Expansion
```

## 3.2 V2 Adds Only Local Semantic Acceleration

S5.v2 adds:

- local embedding generation for supported artifact classes
- `sqlite-vec` similarity search on the same SQLite store used by V1
- hybrid ranking or reranking over V1 candidate sets
- explicit enabled or disabled state with surfaced fallback behavior

## 3.3 V2 Does Not Add A Second Runtime Stack

S5.v2 does NOT add:

- remote embedding providers as part of the default path
- external vector databases
- a mandatory graph runtime
- a second source of truth
- a separate retrieval service or daemon

---

# 4. Retrieval Inputs

The local semantic accelerator MAY generate embeddings for:

- Markdown documentation
- review findings
- verification evidence
- source-file chunks
- runtime traces
- compatible Canon artifacts

The vector layer should only index artifact classes that already have a clear
provenance and an explainable recovery path back to the V1 baseline.

---

# 5. Configuration Model

S5.v2 is not a second mutually exclusive mode.

It is an additive accelerator on top of S5 V1.

The intended configuration states are:

```text
semantic_acceleration = disabled
semantic_acceleration = local
```

Rules:

- `disabled` remains the default until the workspace enables local semantic acceleration
- `local` enables `sqlite-vec` and local embeddings on the same workspace-local index
- if local semantic acceleration is unavailable, Boundline falls back explicitly to the S5 V1 path

Remote semantic providers remain deferred beyond this specification.

---

# 6. Explainability Requirements

When S5.v2 is enabled, the runtime MUST explain:

- why a vector-backed match was selected
- which source artifact produced the match
- whether the match reranked or merely expanded an existing V1 candidate set
- why a vector-backed candidate was rejected or downgraded
- when the runtime fell back from V2 to V1

S5.v2 is invalid if it produces opaque similarity matches that cannot be
explained on `inspect`, `status`, or trace surfaces.

---

# 7. Sequencing Relative To S6 And S7

The intended roadmap order is:

```text
S5 V1
→ S5.v2
→ S6
→ S7
```

## Why S5.v2 Comes Before S6

S6 reasoning profiles benefit from richer, semantically expanded evidence and
implementation-precedent recall.

Without S5.v2, S6 would have to rely more heavily on raw provider behavior or
shallow keyword context.

## Why S5.v2 Comes Before S7

S7 is the assistant-facing affordance layer.

It will surface retrieval, evidence, risk, and reasoning cues to the user.

That layer becomes more credible when semantic matches and fallback behavior
already exist as stable runtime surfaces.

S7 may still prototype on top of S5 V1, but the full roadmap sequence should
treat S5.v2 as the semantic-retrieval prerequisite for the deeper S7 surface.

---

# 8. Acceptance Criteria

The implementation is complete when:

- Boundline can perform optional local semantic similarity on the same SQLite
  store used by S5 V1
- disabling or losing `sqlite-vec` falls back explicitly to the V1 retrieval
  path
- semantic acceleration does not change the authority order
- local semantic acceleration does not require any external service
- runtime surfaces explain why vector-backed matches were selected or rejected
- S6 and S7 can depend on the resulting semantic surfaces without redefining
  retrieval authority
