# Advanced Context Intelligence — Retrieval Technology Evaluation

## Status

Sidecar Design Note

## Purpose

This document evaluates the retrieval and graph technologies considered for the `Advanced Context Intelligence` specification.

The goal is to preserve Boundline's:

- local-first architecture
- cross-platform distribution simplicity
- embedded runtime model
- deterministic installation experience
- lightweight operational footprint

This document is intentionally implementation-oriented and complements the main specification.

---

# 1. Architectural Principle

Boundline is a local-first delivery runtime.

The retrieval subsystem MUST NOT:
- require mandatory infrastructure services
- require distributed deployment
- introduce operational complexity
- compromise offline usage
- compromise deterministic local execution

The retrieval layer exists as:
- an acceleration layer
- a context discovery layer
- an explainability enhancement layer

It MUST NOT become authoritative state.

Authoritative state remains:
- Canon project memory
- runtime manifests
- runtime traces
- workspace configuration
- structured runtime indexes

---

# 2. Recommended Direction

## Recommended Initial Stack

```text
V1:
SQLite + FTS5 + structured indexes

V2:
SQLite + sqlite-vec

Optional:
LanceDB
Kùzu
External providers
```

# 3. SQLite As Foundation

## Why SQLite

SQLite is the preferred foundational storage layer because it is:

- embedded
- serverless
- cross-platform
- deterministic
- operationally simple
- already mature in Rust ecosystems
- compatible with local-first CLI tooling

SQLite allows:
- structured metadata
- JSON storage
- relationship modeling
- full-text search
- local persistence
- single-file distribution

---

## Proposed Storage Path

```text
.boundline/index/boundline.db
```

## Initial Tables

Examples:


# 4. Full-Text Search (FTS5)

## Purpose

FTS5 provides:
- keyword retrieval
- phrase matching
- ranking
- lightweight retrieval
- deterministic explainability

This is sufficient for:
- V1 retrieval
- artifact lookup
- standards lookup
- Canon packet lookup
- review finding search

---

## Why FTS First

FTS retrieval:
- is easier to explain
- is deterministic
- requires no embeddings
- avoids model/provider dependencies
- works fully offline

This aligns with Boundline's local-first philosophy.

---

# 5. sqlite-vec

## Purpose

sqlite-vec enables vector similarity search inside SQLite.

This preserves:
- embedded deployment
- single-file storage
- local execution
- zero-server architecture

---

## Why sqlite-vec

sqlite-vec is preferred because:
- it extends SQLite directly
- it avoids introducing separate infrastructure
- it fits the Boundline runtime philosophy
- it minimizes operational complexity

---

## Intended Usage

sqlite-vec SHOULD support:
- similar review retrieval
- similar architecture decisions
- similar implementation discovery
- semantic Canon packet matching
- semantic evidence retrieval

---

## Important Constraint

Vector retrieval MUST remain:
- optional
- explainable
- secondary to structured retrieval

Vector similarity MUST NOT become the primary source of truth.

---

# 6. LanceDB

## Status

Optional Provider

---

## When To Use

LanceDB MAY become useful when:
- repositories become very large
- vector volume grows significantly
- retrieval latency becomes important
- hybrid vector pipelines become more complex

---

## Why Not Default

LanceDB is intentionally NOT the default because:
- SQLite is simpler
- SQLite is more universal
- SQLite preserves simpler packaging
- SQLite minimizes release risk

---

# 7. Qdrant

## Status

Optional External Provider

---

## Intended Use

Qdrant MAY support:
- enterprise shared indexes
- multi-repository retrieval
- centralized retrieval infrastructure
- distributed retrieval

---

## Why Not Core

Qdrant is not suitable as the default because:
- it requires a service/runtime
- it increases operational burden
- it weakens local-first guarantees
- it complicates installation

Boundline core SHOULD remain embedded-first.

---

# 8. Neo4j

## Status

Not Recommended As Default

---

## Reasoning

Neo4j introduces:
- operational complexity
- external runtime management
- server infrastructure
- deployment burden

This conflicts with:
- local-first execution
- lightweight CLI distribution
- deterministic setup

Neo4j MAY be supported through future provider plugins.

---

# 9. Kùzu

## Status

Preferred Future Embedded Graph Candidate

---

## Why Kùzu

Kùzu is interesting because it is:
- embeddable
- graph-oriented
- lightweight
- Cypher-compatible
- local-first friendly

---

## Potential Future Usage

Kùzu MAY support:
- impact analysis
- invariant propagation
- reviewer inference
- contract relationship traversal
- graph explainability

---

## Why Deferred

Graph infrastructure is intentionally deferred because:
- structured manifests solve most V1 problems
- graph reasoning should build on stable runtime metadata
- retrieval quality matters more than graph complexity initially

---

# 10. SurrealDB

## Status

Not Recommended

---

## Reasoning

SurrealDB introduces:
- unnecessary architectural complexity
- broader infrastructure scope
- less deterministic deployment behavior
- features not currently required by Boundline

The runtime does not currently need:
- distributed graph-document infrastructure
- multi-tenant data systems
- cloud-native persistence layers

---

# 11. Embedding Providers

## Supported Modes

Boundline SHOULD support:

```text
disabled
local
remote
```

---

## Default

Default configuration SHOULD be:

```toml
[embeddings]
enabled = false
```

---

## Local Embeddings

Possible future local providers:
- ONNX
- Candle
- llama.cpp
- local transformer runtimes

Local embeddings preserve:
- offline execution
- privacy
- deterministic behavior

---

## Remote Embeddings

Possible future remote providers:
- OpenAI
- Azure OpenAI
- Gemini
- Anthropic-compatible providers

---

## Security Constraint

Remote embeddings MUST remain opt-in because:
- source code may leave the local machine
- Canon artifacts may contain sensitive information
- enterprise environments may forbid external transmission

---

# 12. Retrieval Philosophy

## Structured Before Semantic

Boundline MUST prioritize:

```text
structured manifests
→ runtime indexes
→ Canon project memory
→ workspace overrides
→ semantic retrieval
```

---

## Explainability Requirement

The runtime MUST explain:
- why an artifact was retrieved
- why a relation exists
- why a similarity match occurred
- why a reviewer was inferred

# 14. Final Recommendation

The recommended default architecture for Boundline is:

```text
SQLite
+
FTS5
+
optional sqlite-vec
```

This combination provides:
- embedded deployment
- local-first guarantees
- explainability
- deterministic packaging
- low operational burden
- cross-platform compatibility

Advanced retrieval providers MUST remain optional extensions rather than core runtime requirements.
