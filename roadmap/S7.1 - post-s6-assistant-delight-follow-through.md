# Post-S6 Assistant Delight Follow-Through

## Status

Future Follow-Through Specification

## Priority

After S6

## Relationship To Delivered Work

The primary assistant delight layer is already delivered in Boundline `0.60.0`
through [`060-assistant-delight-layer`](../specs/060-assistant-delight-layer/spec.md)
and Canon's provider-side `057-s7-delight-provider` specification.

This roadmap item exists only for the remaining follow-through that was not
required for the first delivery slice and should wait until advanced
reasoning-profile support is real runtime capability rather than a future
dependency.

This specification MUST NOT start before S6 is implemented well enough to make
reasoning-profile-aware delight behavior credible.

---

# 1. Outcome

Boundline extends the delivered assistant delight layer with the post-S6 depth
that depends on mature reasoning profiles rather than only the current runtime,
trace, and Canon-input surfaces.

The goal is not to re-ship the existing delight commands.

The goal is to deepen them only where S6 can materially improve judgment,
challenge quality, and inspectability.

---

# 2. Scope

This follow-through is limited to work that was intentionally deferred after the
first assistant delight delivery.

## 2.1 Reasoning-Profile-Aware Explanations

When S6 profiles are active, Boundline should let `challenge`, `hidden-impact`,
and `explain-plan` disclose:

- which reasoning profile is active
- why that profile was chosen
- where profile-specific challenge or synthesis changed the answer
- what fallback was used when profile support is unavailable

## 2.2 Inspect Surface Closure

The first delight delivery already made `inspect` more human-facing for risk,
assumptions, evidence, blockers, and next.

This follow-through should close the remaining roadmap-level inspect surfaces so
operators can navigate:

- `inspect context`
- `inspect council`
- `inspect timeline`

with the same source attribution, fallback disclosure, and operator-facing
clarity already expected from the delivered explanation views.

## 2.3 Host-Surface Follow-Through

The first delivery shipped full command assets for Claude, Codex, and Copilot,
plus compact bootstrap guidance for Cursor and Gemini where richer packaging was
not yet the primary target.

This follow-through should decide whether those hosts can credibly support:

- equivalent contextual cognitive commands
- richer repo-local package affordances
- explicit fallback paths when parity is still not worth the complexity

## 2.4 Product Feedback Signals

If the delight layer is supposed to feel immediately useful, Boundline should be
able to measure at least lightweight operator-facing signals for:

- time to first useful answer
- evidence-attributed explanation rate
- command palette noise
- next-action acceptance or override rate

---

# 3. Non-Goals

This follow-through does NOT:

- redefine the already delivered delight command set
- add new Canon provider artifact classes
- replace governance or stop semantics
- create a second assistant runtime
- reopen the completed `060-assistant-delight-layer` delivery slice

---

# 4. Acceptance Criteria

The follow-through is complete when:

- reasoning-profile-aware delight output is available only when S6 capability is
  actually active and remains explicitly disclosed
- `inspect context`, `inspect council`, and `inspect timeline` are available as
  human-facing operator surfaces without falling back to raw trace reading
- host-surface parity decisions for Cursor and Gemini are explicit, delivered,
  or intentionally rejected with bounded fallback guidance
- the assistant delight layer has at least lightweight measurable feedback loops
  for usefulness and command-surface clarity

---

# 5. Dependency Rule

S7.1 depends on S6 rather than preceding it.

If S6 reasoning profiles remain future-facing, this document stays future-facing
too.
