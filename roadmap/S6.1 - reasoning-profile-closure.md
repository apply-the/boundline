# Reasoning Profile Closure

## Status

Future Follow-Through Specification

## Priority

After `061-reasoning-profile-contracts`

## Relationship To Delivered Work

Boundline `0.61.0` delivered the first S6 runtime contract layer through
[`061-reasoning-profile-contracts`](../specs/061-reasoning-profile-contracts/spec.md).
That slice made reasoning profiles real runtime state inside the existing
session and governance loop, added Canon posture compatibility, and exposed
bounded reasoning traces and operator-facing projections.

What it did **not** close uniformly is every S6 profile from the roadmap as a
concrete end-to-end profile with symmetric activation, positive-path evidence,
and operator-visible closure.

This document exists to finish that closure work explicitly instead of treating
those gaps as implicit future cleanup.

When the next implementation spec is cut, it should take the next available
feature number (`062-*`) and use this document as its carry-forward input.

## Carry-Forward Into `062-*`

The next implementation spec should pull the following audited items from this
document:

- close `independent_pair_review` with a successful converged or adjudicated
  positive-path scenario, not only the blocked independence path
- close `heterogeneous_security_review` with a dedicated end-to-end activation,
  `status`, `inspect`, and confidence story
- close `bounded_reflexion` with a real runtime activation path instead of only
  fixture and unit-trace coverage
- decide whether debate remains bounded substrate or becomes a shipped concrete
  profile id, then validate that decision end-to-end
- decide whether adjudication remains a shared reasoning primitive or becomes a
  shipped standalone profile, then validate that decision end-to-end
- keep runtime-contract, validation-report, and roadmap language aligned so
  concrete shipped profiles are not conflated with family-level or trace-level
  substrate

---

# 1. Outcome

Boundline extends the delivered S6 substrate into a fully closed first-wave
profile set where every profile claimed as implemented has:

- one concrete runtime profile shape or an explicit declaration that it is a
  family-level primitive rather than a shipped profile
- bounded activation inside the normal session-native loop
- positive-path and degraded-path validation where applicable
- explicit inspect, trace, and confidence behavior
- documented acceptance boundaries

# 2. Current Classification

Implemented as `bounded_self_consistency`.

Current delivered evidence includes:
- concrete profile id and family in the runtime vocabulary
- stage activation through the existing session runtime
- status and inspect projection

- confidence and reasoning-trace projection

This is the closest profile to a fully closed first-slice implementation.

## 2.2 Partially Closed In `061`

- independence checks

- adjudication mode wiring

What is still missing:
- a successful converged or adjudicated positive-path scenario that proves the
  profile works beyond the blocked topology case

What is still missing:
- a dedicated end-to-end activation scenario
- status and inspect evidence for a real heterogeneous execution
- explicit confidence and next-action closure for the heterogeneous path

### Reflexion

Represented by the `bounded_reflexion` profile.

What is already real:
- concrete profile id and family mapping
- reflexion budget rules
- fixture coverage
- unit trace coverage for reflexion iterations

What is still missing:
- an end-to-end runtime activation scenario
- explicit critique or revise participant topology exercised through the normal
  workflow

## 2.3 Substrate Present But Not Closed As A Shipped Profile In `061`

### Debate

What is already real:
- `debate_enabled` family vocabulary
- budget validation for debate rounds
- debate-round trace events
- roadmap-level runtime requirements carried into the spec and trace contract

What is still missing:
- a shipped V1 concrete debate profile id
- end-to-end activation for a debate profile
- explicit participant roles and stagnation handling proven through status,
  inspect, and trace evidence

### Adjudication

What is already real:

What is still missing:

  reasoning primitive
- an end-to-end reasoning-profile scenario where disagreement resolves through
  explicit adjudication and is then surfaced through the normal operator views

# 3. Scope

This follow-through is limited to the remaining closure work required to make
S6 claims precise and honest.

## 3.1 Concrete Profile Closure

For each profile still partially closed, Boundline should add:
- one explicit activation scenario
- one positive-path validation story where applicable
- one degraded, blocked, or interrupted story where applicable
- one operator-visible explanation path across `run`, `status`, and `inspect`

## 3.2 Debate Closure

If debate is meant to be part of the shipped S6 wave, Boundline should define:
- at least one concrete V1 debate profile
- explicit debate participant roles
- bounded stagnation and low-value continuation handling
- trace and inspect evidence that debate is not just vocabulary

If this cannot be done credibly, debate should remain documented as supported
substrate rather than claimed as a closed shipped profile.

## 3.3 Adjudication Closure

Boundline should make explicit whether adjudication is:
- a shipped standalone reasoning profile, or
- a shared reasoning primitive used by other profiles

The answer must be documented and validated, not inferred.

## 3.4 Profile Claim Hygiene

The runtime contract, validation report, and roadmap should distinguish between:
- concrete shipped profile ids
- family-level primitives
- trace-only or budget-only substrate

---

# 4. Non-Goals

This follow-through does NOT:
- introduce a second orchestration system
- reopen the Canon posture contract boundary
- add unbounded swarm execution
- turn every future reasoning pattern into a V1 shipped profile
- redefine S3 or S4 governance ownership

---

# 5. Acceptance Criteria

The follow-through is complete when:

- every S6 profile claim is classified explicitly as concrete shipped profile,
  family-level primitive, or deferred work
- `independent_pair_review`, `heterogeneous_security_review`, and
  `bounded_reflexion` each have credible end-to-end evidence or are explicitly
  narrowed
- debate is either delivered as a concrete bounded profile or explicitly left as
  substrate rather than implied implementation
- adjudication is explicitly documented as profile or primitive, and the chosen
  status is backed by runtime evidence
- `status`, `inspect`, and trace behavior remain aligned with those claims

---

S6.1 depends on the delivered `061-reasoning-profile-contracts` slice.

S7.1 should assume mature reasoning-profile support only after this closure work
makes the profile set and its evidence explicit.
